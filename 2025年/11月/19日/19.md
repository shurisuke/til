# Figma関連
- Auto Layout の解除は Shift + A ではできない（代わりに「Remove Auto Layout」から操作する）。
- グループとフレームの違いを理解。枠を編集したい時はグループではなくフレームを使うと柔軟。
- Figmaで画面構成を管理するには「セクション」機能が便利（画面一覧にまとめるときなど）。
- 範囲選択や、複数選択して操作する機能を活用することでレイアウト効率が上がる。


# アプリ構想のメモ
- スポット登録時、Google Place API の photo_reference を保存し、後で画像URLを動的に生成して表示する構成は正しい。
- 地図上でスポットを選択し、それをプランに追加 → プラン作成メニューに反映、という流れはUI/UXとして自然。
- 「タグ」と「メモ」の役割の違いに注意。将来的にタグでの検索機能を見据えると、タグを別管理するのが望ましい。
- プランを公開する際は、ユーザー名を伏せる形にして、都道府県・タグ・スポット構成などで共有できるようにすると個人情報漏洩のリスクを抑えられる。


# 機能実装する時に必要なコードメモ
- ユーザー情報の編集において、更新失敗時は render を使って編集画面を再表示することで、エラーメッセージをそのまま表示できる。例：render :edit。
- 編集画面に遷移させるボタン名は、単に「編集」よりも「設定を変更する」や「プロフィールを編集」などの具体的な文言にした方が親切。
- 「カレントユーザー（current_user）」はログイン中のユーザーを指す。アプリ内でログイン状態のチェックや、ユーザーの関連情報を取得するのに使う。
- コントローラーでのリダイレクト先に redirect_to spots_path のように書くことで、更新後に一覧画面などに戻す遷移を明示できる。
- フラッシュメッセージの使い方を理解。flash[:notice] や flash[:alert] を使って、処理成功や失敗のフィードバックを表示できる。
